# WhareScore UX Audit — 2026-04-12

**Tested:** desktop 1440x900 via Chrome automation against `https://wharescore.co.nz`.
**Test address:** 54B Norway Street, Aro Valley, Wellington (apartment/townhouse in
Wellington SA2 "Kelburn" — surfaces a known suburb/SA2 naming mismatch).
**Persona:** buyer by default, some checks in renter mode.
**Method:** walked the live site as a first-time user would — landing page, map
interaction (zoom, click a pin, open legend, layer picker), property report
scroll-through for every section, paywall modal, and spot checks on static pages.

Only items that were actually fixed in this pass are listed. Deferred work from
earlier audits (map pin clustering, etc.) is tracked in `UX-AUDIT-2026-04-11.md`.

## Critical — contradictions users see immediately

| # | Status | Issue | Where | Fix |
|---|---|---|---|---|
| A1 | ✅ | Buyer checklist said **"No contaminated sites found nearby"** while the Planning section showed **"Contaminated sites nearby: 181"** for the same property. | `BuyerDueDiligence` read `hazards.contamination_count` which `transformReport` always set to `null` ("moved to planning" comment), so the checklist was stuck on the "zero" branch regardless of actual data. | Transform now mirrors `planning.contamination_count` onto `hazards.contamination_count` after both transforms run. All four call sites (`BuyerDueDiligence`, `BuyerChecklistContent`, `ActionCard`, `FindingCard`) also fall back to `planning` explicitly. Copy unified on "HAIL sites within 500m" so the checklist, finding and planning row now all say the same thing. |
| A2 | ✅ | **"Insurance Risk: Low Risk — Standard insurance likely available at normal premiums"** appeared right next to 5 indicator cards reading "Wind High risk / Slope Failure High risk / Air Quality High risk / Water Quality High risk / Crime High risk". | `InsuranceRiskCard` only considers 7 insurer-relevant hazards (flood, EPB, slope, tsunami, coastal, wildfire, landslide). The fact that wind/air/crime aren't part of the insurance assessment wasn't communicated, so users read the card as contradicting the indicator list. | Reworded the green-level message to spell out exactly which hazards insurers load for and to call out that wind/air/crime don't typically change premiums. Subtitle changed from "Based on N hazard indicators" to "Checks N insurer-relevant hazards" so users know the scope up front. |
| A3 | ✅ | Report header "Aro Valley, Wellington" vs footer social proof **"28 reports for Kelburn this month"** and suburb info block titled **"Kelburn — Suburb information"**. | `SocialProof` and `ReportCTABanner` were reading `report.address.sa2_name` which is a StatsNZ SA2 label, not the user-facing suburb. `AISummaryCard` correctly points at the SA2 because the narrative is written per SA2. | `SocialProof` + `ReportCTABanner` now prefer `address.suburb`, falling back to `sa2_name`. `AISummaryCard` intentionally keeps `sa2_name` because its body content is generated per SA2 — a comment in `PropertyReport.tsx` documents why the two diverge on purpose. |
| A4 | ✅ | Rental market chip showed **"1yr -31.3%"** — an absurd outlier that passed straight through from a small-sample StatsNZ series. Small SA2s with sparse bond data can produce 30+% moves that are pure signal noise, not real market movement. | Backend trend series are fed directly into `market.trend.cagr_1yr` with no sanity check. | `transformReport.transformMarket` now runs every CAGR through `sanitiseCagr` — anything outside ±25% for 1yr and ±15% for 5/10yr is nulled out rather than displayed. Those ranges cover every genuine Auckland/Wellington movement in the past 20 years with generous margin. |
| A5 | ✅ | "7 things to investigate" list included **"Schools: 100/100"**. Schools score 100 when there are **no** usable schools nearby — technically correct under the risk framing (100 = worst) but reading as "this property scored 100% on schools" to users. Same property's Liveability section showed 4 nearby schools, so the 100/100 was a broken pipeline anyway. | `school_quality_score` in `risk_score.py` only scored schools with an EQI value joined. When the EQI feed isn't joined, every school was skipped, quality_points came out at 0, and the final score collapsed to 100. | Added a proximity-based fallback in `school_quality_score`: when schools exist but none have EQI, use distance to nearest + count-bonus to produce a sensible score. KeyTakeaways also now prints "Low / Moderate / High risk" labels instead of raw `N/100` values — so even when scoring is right, users never see "17/100" and misread it as "17% fair". |

## Critical — data quality leak

| # | Status | Issue | Where | Fix |
|---|---|---|---|---|
| A6 | ✅ | District Plan card showed **"Category: Zone"** — a literal useless string. | WCC's ArcGIS feed returns `"Zone"` as the `Category` value for MRZ zones, which passed straight through to `district_plan_zones.category`. | Two layers of defence: (1) `load_district_plan_zones` now calls `_derive_zone_category` as a fallback when the raw Category is empty, equals the zone name, or is the literal "Zone" — so data reloads produce "Residential" / "Business" / etc.; (2) the frontend `PlanningSection` also filters out `/^zone$/i` defensively so existing DB rows render cleanly without a backfill. |

## Critical — form UX

| # | Status | Issue | Where | Fix |
|---|---|---|---|---|
| A7 | ✅ | Opening the Generate-Report modal **immediately** surfaced a red "Select the number of bedrooms to continue" banner with nothing the user had done yet. | `ReportConfirmModal` bound the banner directly to `!isReady`, which is true from first render because the buyer bedroom default is empty. | Added `attemptedSubmit` state which resets on modal open and only flips to `true` when the user clicks the Generate button. Button is now always clickable (previously `disabled`) so click → validate → show error → highlight fields. Primary button uses dimmed styling when not ready so visual feedback is preserved. |
| A8 | ✅ | Persona toggle buttons had no `aria-pressed` / `aria-selected` — screen readers couldn't announce which persona was active. | `PersonaToggle`. | Added `role="tablist"` on the wrapper with a descriptive label; each button is a `role="tab"` with both `aria-selected` and `aria-pressed` so NVDA/JAWS and Safari VO both see the state correctly. |

## High — wording and labels that mislead

| # | Status | Issue | Where | Fix |
|---|---|---|---|---|
| A9 | ✅ | BuyerSnapshot headline said **"Elevated risk — due diligence critical"** while the top-of-report badge said **"Low Risk 39"**. Two different subsystems both producing an overall verdict, worded so they contradict. | `BuyerSnapshot.VERDICT_CONFIG` labels were written as if they were risk-score labels. | Rewrote all four verdict headlines to focus on what the buyer should DO, not on re-scoring: "Strong fundamentals — standard checks apply", "A few areas to look closer at", "Worth extra due diligence before offering", "Several hazards flagged — thorough review needed". No more word-level collision with the score badge. |
| A10 | ✅ | Header line showed the full address "54B Norway Street, Aro Valley, Wellington" and then a redundant "Aro Valley, Wellington" directly underneath. | `PropertySummaryCard` always rendered `{suburb}, {city}` as a subheading. | Subheading is now conditional: if the `full_address` already contains both the suburb and city (which is typical for LINZ addresses), the redundant line is suppressed entirely. If one is missing, only the missing parts are rendered. |
| A11 | ✅ | Walkability pills showed raw scores like **"Nearby amenities: 25"** / **"Transit access: 25"** / **"Schools: 12"**. Users read them as counts (25 amenities!) when they're actually contributions out of category caps (25/25, 12/15, etc). | `WalkabilityScore`. | Pills now show `{score}/{maxScore}` with a hover tooltip explaining "contributes X out of Y points to the overall walkability score". |
| A12 | ✅ | "daily-life" chip said just **"13 stops"** with no buffer context, while the nearby Transport section said **"Transit stops within 800m"**. Users couldn't tell whether it was the same count or a different radius. | `QuestionSummary`. | Daily-life chip now reads `{N} stops (400m)` so the radius is explicit, matching the underlying `transit_count` field which is the 400m count. |
| A13 | ✅ | **"Earthquake-prone building — Not listed"** row in the property checklist. Reads like "this is an earthquake-prone building, not listed yet" to a non-technical user. | `EpbListedItem`. | Row now asks a direct question — **"This building on the EPB register?"** — with a `"Yes — listed"` / `"No"` answer. Same data, unambiguous reading. |
| A14 | ✅ | AreaEventTeaser headline **"43 significant events near this property"**. In Wellington most of those 43 are minor background seismic events from GeoNet — "significant" was hyperbole. | `AreaEventTeaser`. | Headline now reads "43 recent events tracked near this property" — accurate (we do track them) and not alarmist. |

## Medium — polish and noise

| # | Status | Issue | Where | Fix |
|---|---|---|---|---|
| A15 | ✅ | Signed-in users still saw the **"Free property report — sign in for the full version"** banner. Dead space for anyone who's already signed in. | `BetaBanner` had no session awareness. | Banner now reads `useSession()` and returns `null` when `status === 'authenticated'`. |
| A16 | ✅ | Clicking anywhere inside the map Legend wrapper could fall through to the map underneath (I hit a property selection by tapping the legend expand area). | `MapLegend` relied purely on z-index. MapLibre's click handler fires on the map div; without an explicit stopPropagation the click bubbled through the legend. | Wrapper and button both call `stopPropagation` on `onMouseDown`, `onClick`, `onTouchStart` so legend interactions never reach the map. |
